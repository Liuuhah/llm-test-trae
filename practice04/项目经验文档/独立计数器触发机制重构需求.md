# 5W 提取触发机制重构需求文档

## 📋 任务概述

将当前的**基于 chat_history 索引的累积提取机制**重构为**基于独立计数器的触发机制**，解决压缩操作导致索引失效的问题。

---

## 🎯 核心问题

### **当前方案的问题**

**现有逻辑：**
```python
self.last_failed_start_index = None  # 记录起始索引

# 提取时：
recent_messages = self.chat_history[self.last_failed_start_index:]

# 问题：
# 1. 压缩会删除历史消息，导致索引失效
# 2. 需要复杂的索引调整逻辑
# 3. 容易出错（如出现 index=2 的异常情况）
```

**具体案例：**
```
第 1-5 轮 → 提取失败，记录 start_index=0
  ↓
触发压缩 → 删除前 7 条消息
  ↓
chat_history 被截断，但 start_index 仍然是 0
  ↓
下次提取时：chat_history[0:] 指向错误的消息
  ↓
结果：提取范围混乱，可能只提取到部分消息
```

---

## 💡 新方案设计

### **核心思路：使用独立计数器**

不再依赖 `chat_history` 的索引，而是维护一个**独立的消息计数器**，与压缩操作完全解耦。

### **关键变量**

```python
self.extract_message_counter = 0  # 5W 提取消息计数器
```

### **工作流程**

#### **1. 用户发送消息时累加计数器**

```python
# 在用户输入处理后、添加到 chat_history 之前
self.extract_message_counter += 1
```

#### **2. AI 回复完成后检查是否触发**

```python
# 在 AI 回复添加到 chat_history 之后
if self.extract_message_counter > 0 and self.extract_message_counter % self.extract_interval == 0:
    # 触发 5W 提取
    self._extract_5w_info()
```

#### **3. 提取成功后重置计数器**

```python
if status == "success":
    # 提取成功，重置计数器
    self.extract_message_counter = 0
    print(f"[关键信息提取] ✅ 提取成功，计数器已重置")
else:
    # 提取失败，计数器继续累加
    print(f"[关键信息提取] ⚠️ 提取失败，计数器保持为 {self.extract_message_counter}")
```

### **修改 3：修改触发条件判断**

**位置：** `run()` 方法中 AI 回复后的提取/压缩逻辑

**当前代码：**
```python
if self._should_compress():
    # 触发提取和压缩
```

**修改为：**
```python
# 检查是否达到提取间隔
if self.extract_message_counter > 0 and self.extract_message_counter % self.extract_interval == 0:
    print("\n[系统触发] 检测到对话达到提取阈值，开始执行归档与清理...")
    
    # 1. 先执行关键信息提取 (5W)
    if self.auto_extract_enabled and not self.skip_next_extract:
        print("[系统触发] 步骤 1/2: 正在执行 5W 关键信息提取...")
        self._extract_5w_info()
    elif self.skip_next_extract:
        self.skip_next_extract = False
        print("[系统触发] 用户已跳过本次 5W 提取")
    
    # 2. 后执行上下文压缩（仅在满足压缩条件时）
    if self._should_compress():
        if self.auto_compress_enabled and not self.skip_next_compress:
            print("\n[系统触发] 步骤 2/2: 正在执行上下文压缩...")
            self._compress_chat_history()
        elif self.skip_next_compress:
            self.skip_next_compress = False
            print("[系统触发] 用户已跳过本次上下文压缩")
else:
    # 未达到提取间隔，不执行任何操作
    pass
```

**说明：**
- 提取触发条件改为：`extract_message_counter % extract_interval == 0`
- 压缩仍然使用原有的 `_should_compress()` 判断
- 提取和压缩可以独立触发（不一定同时发生）

## ⚠️ 注意事项

### **核心注意事项**

1. **计数器只在用户发送消息时累加**
   - AI 回复不计入计数器
   - 工具调用不计入计数器

2. **提取成功后必须重置计数器**
   - 否则会导致频繁触发

3. **提取失败时不要重置计数器**
   - 让它继续累加，实现累积效果

4. **压缩操作不影响计数器**
   - 这是新方案的核心优势

5. **保持向后兼容**
   - 不影响现有的压缩逻辑
   - 不影响工具调用
   - 不影响日志保存

---

### **⚠️ 重要：提取和压缩不同步的潜在问题**

**当前设计：**
- 提取触发条件：`extract_message_counter % 5 == 0`（第 5、10、15... 轮）
- 压缩触发条件：`rounds > 5`（第 6、7、8... 轮才满足）

**导致的结果：**
- **第 5 轮**：只触发提取，不触发压缩
- **第 10 轮**：同时触发提取和压缩
- **第 15 轮**：同时触发提取和压缩

---

#### **可能的问题 1：chat_history 过长**

**场景：**
```
第 5 轮 → 提取成功，但 chat_history 仍然是 10 条消息（未压缩）
第 6-9 轮 → 继续累积，chat_history 变成 18 条消息
第 10 轮 → 才触发压缩
```

**风险：**
- 在第 5-9 轮期间，chat_history 可能超过 `max_context_tokens` 限制
- LLM 处理长上下文时性能下降
- 可能触发 token 超限错误

**解决方案：**
- 监控 `_get_context_tokens()` 的值
- 如果接近上限，考虑将压缩条件改为 `rounds >= 5`

---

#### **可能的问题 2：提取范围与压缩范围不一致**

**场景：**
```
第 5 轮提取：提取最近 10 条消息（第 1-5 轮）
第 6-9 轮：用户继续对话，chat_history 变成 18 条
第 10 轮压缩：压缩前 14 条，保留后 4 条
```

**风险：**
- 第 5 轮提取的内容可能在第 10 轮压缩时被删除
- 如果第 5 轮提取失败，累积的消息可能在压缩时丢失

**示例：**
```
第 1-5 轮 → 提取失败（counter=5）
第 6-9 轮 → 继续对话（counter=9）
第 10 轮 → 提取 + 压缩
  - 提取：最近 10 条（第 6-10 轮）
  - 压缩：删除前 14 条（包括第 1-5 轮）
  - 结果：第 1-5 轮的信息永久丢失！
```

**解决方案：**
- 在压缩前检查 `extract_message_counter > extract_interval`
- 如果大于，说明有未成功提取的历史，先提取再压缩
- 或者将压缩条件改为 `rounds >= 5`，确保同步

---

#### **可能的问题 3：日志记录不连续**

**场景：**
```
第 5 轮 → 提取成功，保存到日志
第 6-9 轮 → 没有提取，日志中没有记录
第 10 轮 → 提取成功，保存到日志
```

**风险：**
- 日志中缺少第 6-9 轮的记录
- 搜索历史时可能找不到中间阶段的对话

**影响程度：**
- ⚠️ **中等**：如果第 6-9 轮没有重要信息，影响不大
- ❌ **严重**：如果第 6-9 轮有关键信息，会丢失

**解决方案：**
- 接受当前的设计（每 5 轮记录一次）
- 或者降低 `extract_interval` 到 3，提高记录频率

---

#### **可能的问题 4：用户体验不一致**

**场景：**
```
用户发送第 5 条消息 → AI 回复 → 触发提取（用户看到延迟）
用户发送第 6-9 条消息 → AI 回复 → 无额外操作（正常速度）
用户发送第 10 条消息 → AI 回复 → 触发提取 + 压缩（用户看到更长延迟）
```

**风险：**
- 用户在第 5 轮和第 10 轮感受到不同的响应速度
- 可能造成困惑：“为什么有时候慢，有时候快？”

**解决方案：**
- 在 UI 上显示“正在归档对话...”提示
- 或者将提取和压缩改为异步执行（不阻塞用户）
